# PsyClaw 类人记忆系统设计(三分类 + 遗忘 + 睡眠 + 冲突)

> 蓝图文档(2026-07-14,与用户共同定稿)。实现分四轮:feat-114 语义记忆+冲突
> 协议 · feat-115 统一衰减生命周期 · feat-116 睡眠整合 · feat-117 发散检索。

## 一、三分类与现状映射

| 认知分类 | 职责 | psyclaw 载体 | 现状 |
|---|---|---|---|
| **情景记忆** episodic | 发生了什么(原文,ground truth) | archive(SQLite+FTS5 全量轮次)+ recall 双通道门控 | ✅ 已有 |
| **程序性记忆** procedural | 怎么做(坑与打法) | lessons 卡(环境坑,pending→active→归档)+ habits 惯性;**缺**:成功路径 playbook(只从失败学,不从成功学) | 🔶 半有 |
| **语义记忆** semantic | 研究语境下的概念/约定/事实("本项目缺失码=99/-999"、构念定义、变量口径) | **无**(profile 只有几条画像) | ❌ 缺 → feat-114 |

三层关系:情景是原始底片,语义/程序是从底片上蒸馏出的结构化知识,
每张语义/程序卡都带 `source` 指回情景——**冲突时回底片裁决**。

## 二、编码门槛:不是所有信息都值得记

写入触发只认三类高信号(其余任其消逝,情景档案兜底):
- **代价**:命令失败/被质量检查拦下(已有,错误学习);
- **惊讶**:用户纠正("不对,应该是…")、与既有卡矛盾的新信息;
- **频率**:同一决策/说法重复出现(habits 已有计数;语义卡 use_count)。

## 三、遗忘机制(feat-115):激活度衰减 + 分层处置

每张卡(语义/程序)维护激活度:
```
activation = strength × exp(-λ · days_since_last_used)
```
- 被检索命中/再现印证 → last_used 刷新,strength +1(已有强化机制并入);
- activation < 休眠阈值 → **dormant**:不再参与注入/检索,但保留(睡眠期可复活);
- dormant 超长期且 use_count==0 → 删除(先归档快照);
- **情景档案不删原文**(磁盘便宜),只做检索排序的时近加权——遗忘发生在
  "取用权重"而不是"存储"上;
- 方向不对称(故意):升级为长期要过高门槛/人审,降级遗忘由机器自动——错学比忘记贵。

## 四、睡眠机制(feat-116):`psyclaw sleep` 离线整合

触发:手动 `psyclaw sleep` / 会话结束 hook / auto 空闲阶段。四步:
1. **重放**:扫描上次睡眠以来的情景轮次,LLM 蒸馏候选语义事实与
   程序 playbook(含**成功路径**——补"只从失败学"的缺口);无 LLM 走
   规则蒸馏兜底,产物一律 pending 待确认;
2. **合并**:同主题卡聚类,N 条具体 → 1 条原则(把 feat-112 对 lean_core
   的手工合并自动化到卡库);合并保留成员卡指针,可展开;
3. **衰减结算**:批量更新激活度,执行休眠/复活/删除;
4. **联想铺路**(供 feat-117):对语义卡建共现/类比边(复用 kg 模块),
   给发散检索留索引。

## 五、创新任务的记忆支持(feat-117):发散检索

认知依据:REM 期远距联想促进创造性重组。工程对应:
- recall/语义检索加 `mode="diverge"`:除高相关命中外,**额外采样 k 条
  中等相关(0.3~0.6 区间,带随机)** 的卡进上下文——给设计研究/头脑风暴
  类任务制造远距联想材料;
- 常规任务不启用(噪音);由任务意图(设计/brainstorm 关键词)或显式开关触发。

## 六、记忆冲突协议(feat-114)

三类冲突与处置:
| 类型 | 例 | 处置 |
|---|---|---|
| 语义冲突 | "缺失码=99" vs 后来"缺失码=999" | **时近优先生效但降置信**;旧说法进 history 不删;卡标 conflicted,注入时如实带出「⚠ 曾有不同说法」;`/memory conflicts` 列出待人工裁决 |
| 程序冲突 | 两条教训指向相反做法 | 同上;裁决前两条都注入并标注冲突,让模型知情权衡 |
| 记忆 vs 现实 | 卡说没有 python,probe 发现有了 | 已有:probe 证实才归档(feat-059) |

原则:**绝不静默覆盖,绝不静默丢弃**——冲突是信息,不是脏数据;
语义卡的 source 指回情景原文,终审依据永远是底片。
